# gRPC核心概念与实践

## gRPC 工作原理与架构

gRPC（gRPC Remote Procedure Call）是一个高性能、开源的远程过程调用框架。它的核心思想是：让调用远程服务就像调用本地函数一样简单。

- 核心组件：
  - Protocol Buffers (Protobuf)：接口定义语言 (IDL)，用于定义服务和序列化消息。
  - Stub (存根)：在客户端，存根提供与服务器相同的方法，负责将调用参数包装成二进制格式并发送。
  - Server Skeleton (骨架)：在服务端，负责接收请求、反序列化数据并调用实际的业务逻辑。
- 交互流程：
  - 客户端调用本地生成的 Stub 方法。
  - Stub 将请求参数序列化为二进制（Protobuf）。
  - 通过 HTTP/2 传输到服务端。
  - 服务端接收并反序列化，执行逻辑。
  - 结果按原路二进制返回。

## HTTP/2 的核心优势

gRPC 默认运行在 HTTP/2 之上，这是它比传统 REST (HTTP/1.1) 快得多的主要原因：

| **特性**                    | **说明**                                    | **带来的好处**                           |
| --------------------------- | ------------------------------------------- | ---------------------------------------- |
| **二进制分帧**              | 将信息切分为更小的二进制帧，而非文本。      | 解析速度更快，出错率更低。               |
| **多路复用 (Multiplexing)** | 在同一个 TCP 连接上并行发送多个请求和响应。 | 解决了“队头阻塞”问题，极大提升并发能力。 |
| **头部压缩 (HPACK)**        | 压缩重复的 HTTP 头部信息。                  | 减少了网络带宽的浪费。                   |
| **双向流**                  | 允许客户端和服务器同时发送数据。            | 为实时通信和流式传输提供了原生支持。     |

## 四种通信模式对比

这是我们在代码实践中掌握的四种核心模式：

| **模式**                        | **关键词**                          | **特点**                             | **典型应用场景**                   |
| ------------------------------- | ----------------------------------- | ------------------------------------ | ---------------------------------- |
| **简单 RPC (Unary)**            | `(Req) returns (Res)`               | 一发一收，最简单的请求响应模型。     | 登录、获取单个用户信息。           |
| **服务端流 (Server Streaming)** | `(Req) returns (stream Res)`        | 客户端发一次，服务端持续推送。       | 股票行情、实时监控、分片下载。     |
| **客户端流 (Client Streaming)** | `(stream Req) returns (Res)`        | 客户端持续发，服务端最后回一个总结。 | 大文件上传、物联网传感器数据上报。 |
| **双向流 (Bidi Streaming)**     | `(stream Req) returns (stream Res)` | 双方随时互发，全双工通信。           | 实时聊天、协同编辑、在线游戏。     |

##  拦截器（Middleware）机制

拦截器是实现 **AOP（面向切面编程）** 的关键，可以在不侵入业务代码的前提下增强功能。

- **元数据 (Metadata)**：类似于 HTTP Header，用于在客户端和服务端之间传递 Token 等非业务数据。

## 小结

**安全性**：生产环境下通常开启 **TLS/SSL** 加密。

**兼容性**：利用 `UnimplementedGreeterServiceServer` 嵌入结构体，可以保证在 `.proto` 更新时代码不崩溃。

**流结束标识**：在 Go 中，流的结束是通过捕获 `io.EOF` 错误来判断的。
