# 《星星屿》微信小程序技术架构文档

## 1. 项目简介

**星星屿 (xyu)** 是一款个人物品全生命周期管理工具。通过记录购买价格、使用时间及状态，实时计算物品的“日均成本”与“性价比”，辅助用户建立理性的消费观与断舍离习惯。

## 2. 总体架构图

系统采用典型的 **微服务化单体 (Modular Monolith)** 架构，便于前期快速开发，后期弹性扩展。

```
graph TD
    User((用户)) -->|微信小程序| MP[前端: Uniapp]
    MP -->|HTTPS / JWT| Gateway[Nginx 反向代理]
    Gateway -->|Forward| API[Golang 后端服务]
    
    subgraph "Backend Service (Golang)"
        API --> Auth[用户/鉴权模块]
        API --> Item[物品管理模块]
        API --> Calc[价值计算引擎]
        API --> AI[AI 代理模块]
    end
    
    API --> DB[(PostgreSQL)]
    API --> Redis[(Redis - 缓存/限流)]
    AI --> ThirdParty[第三方 AI 供应商]
    API --> OSS[云对象存储 - 图片]
```

## 4. 数据库设计 (Schema)

## 4.1 核心表结构

#### 用户配置表 (`user_configs`)

用于存储用户个人的 AI Key，确保 BYOK (Bring Your Own Key) 模式运行。

| 字段名      | 类型        | 描述                              |
| ----------- | ----------- | --------------------------------- |
| user_id     | varchar(64) | 唯一标识 (OpenID)                 |
| ai_provider | varchar(20) | 供应商 (openai / deepseek / etc.) |
| api_key     | text        | 加密后的 API Key                  |
| base_url    | text        | 代理或自定义 API 地址             |

#### 物品表 (`items`)

| **字段名**  | **类型**      | **描述**                              |
| ----------- | ------------- | ------------------------------------- |
| id          | bigserial     | 主键                                  |
| user_id     | varchar(64)   | 归属用户 (索引)                       |
| name        | varchar(100)  | 物品名称                              |
| category_id | int           | 关联分类                              |
| price       | decimal(12,2) | 购买价格                              |
| purchase_at | date          | 购买日期                              |

| icon_url    | text          | 图片地址                              |
| source_type | smallint      | 来源 (1-预设, 2-拍照, 3-Canvas, 4-AI) |

## 5. 核心逻辑实现

### 5.1 实时价值计算引擎 (Service Layer)

后端不存储计算结果，仅在 `GET /items/:id` 时实时计算：

Days = max(1, (min(Today, RetireDate) - PurchaseDate))

DailyCost = Price / Days

**性价比评价矩阵（示例）：**

- **极高**：DailyCost < ￥1.0 (如：牙刷、耐穿的衣物)
- **高**：￥1.0 < DailyCost < ￥5.0
- **中等**：￥5.0 < DailyCost < ￥20.0
- **低**：￥20.0 < DailyCost < ￥50.0
- **极低**：DailyCost > ￥50.0 (如：吃灰的电子产品)

### 5.2 AI 代理逻辑 (AI Proxy)

1. **解密**：从数据库读取用户 Key，使用服务端环境变量中的 `AES_KEY` 解密。
2. **请求转发**：后端作为中转站，构造符合供应商标准的 Request body。
3. **转存**：获取 AI 生成的图片流，不直接传回前端，而是**先流式上传至 OSS**，然后返回 OSS 地址。
   - *理由*：AI 生成的图片 URL 通常有时效性，转存 OSS 可实现永久访问。

### 5.3 分享海报生成

- **方案**：前端 Canvas 绘制。
- **内容**：物品图片、名称、陪伴时长、日均成本、性价比勋章、小程序码。
- **优化**：后端提供一个 `GET /items/:id/share` 接口，返回格式化好的文案字符串。

## 6. 接口设计 (API)

| **路径**       | **方法** | **功能**                          |
| -------------- | -------- | --------------------------------- |
| `/auth/login`  | POST     | 微信登录，返回 JWT                |
| `/items`       | GET      | 分页获取物品列表 (包含基础计算)   |
| `/items`       | POST     | 创建物品 (逻辑：处理 OSS 上传)    |
| `/items/:id`   | GET      | 详情页 (包含深度价值计算及评价)   |
| `/ai/generate` | POST     | 调用 AI 生成图标 (需校验用户 Key) |
| `/user/config` | PUT      | 更新用户的 AI 供应商配置          |

## 7. 安全与可靠性设计

1. **数据安全**：用户的 `API_KEY` 必须在数据库中**加密存储**，切勿明文。
2. **数据隔离**：GORM 开启 `Scopes` 机制，在所有查询语句中强制带入 `where("user_id = ?", current_user)`。
3. **计算鲁棒性**：处理日期时，需考虑时区问题（统一使用 `UTC+8`），并防御“购买日期晚于当前日期”的非法数据。
4. **成本控制**：由于使用用户自己的 Key，后端主要负责流量透传，需在后端设置 **Request Timeout (如 60s)**，防止占用协程池。

## 8. 部署架构 (Traditional Server)

使用 `docker-compose.yml` 管理：

- **Nginx 容器**：处理 HTTPS 证书及小程序域名的反向代理。
- **Go-App 容器**：后端核心服务。
- **PostgreSQL 容器**：数据持久化，通过 Volume 挂载到宿主机硬盘。
